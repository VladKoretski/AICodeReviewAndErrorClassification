## Пример 1 (Python): команда из пользовательского ввода (риск безопасности)
Ситуация: утилита запускает системную команду. Аргумент приходит от пользователя
```python
import os
def run_backup(folder: str) -> int:    # folder приходит извне (например, из формы)     
cmd = f"tar -czf backup.tgz {folder}"    
    return os.system(cmd)
```
  
### Промпт к ИИ-ревью
```
Ты опытный разраьротчик на пайтон. Функция run_backup запускает архивацию указанной извне папки. Аргумент folder поступает от пользователя (веб-форма/API). Необходимо осуществить проверку кода со следующими критериями:
- безопасность (инъекции, права, обработка ввода)  
- корректность работы с ОС и возврат кодов завершения  
- читаемость и тестируемость  
Сформулируй предложения с учетом следующих ограничений: 
- не предлагай крупный рефакторинг без веской причины  
- не придумывай сторонние библиотеки, используй только стандартную библиотеку Python  
- фокусируйся на критичных рисках и конкретных правках  
Формат ответа: Для каждого замечания укажи в табличном формате:
|Номер замечания|Приоритет: blocker/major/minor | Что не так | Риск | Как исправить|
  
Указанный код:
import os

cmd = f"tar -czf backup.tgz {folder}"

```

Перед запуском проверять права на запись через `os.access(target_dir, os.W_OK)`.|
|5|minor|Использование `os.system` снижает тестируемость и кроссплатформенность|Сложность написания юнит-тестов. `os.system` вызывает `/bin/sh -c`, поведение зависит от ОС, мокать сложно, а сигнатура функции не позволяет изолировать внешние зависимости.|Переход на `subprocess.run` решает проблему. * [blocker] Конкатенация folder напрямую в строку команды создаёт риск Shell-инъекции. Если пользователь передаст ; rm -rf /, команда выполнит произвольный код. Риск: полный компромисс хоста. Как исправить: использовать subprocess.run(["tar", "-czf", "backup.tgz", folder], check=True);
* [major] os.system возвращает только код выхода и не даёт контроля над stdout/stderr. Риск: ошибки архивации остаются незамеченными. Как исправить: заменить на subprocess.run с capture_output=True и логированием;
* [minor] Отсутствует валидация существования директории перед запуском. Риск: tar упадёт с кодом 1, клиент получит ложное ожидание. Как исправить: добавить if not Path(folder).is_dir(): raise ValueError(...);
* [doubtful] Предлагает заменить tar на shutil.make_archive. Риск: меняет формат архива и сжатие, может сломать downstream-процессы. Как исправить: проверить требования к формату бэкапа перед заменой.

**Краткий пример исправленного варианта (только stdlib, без смены сигнатуры):**  
```python
import os
import subprocess

def run_backup(folder: str) -> int:
    # 1. Валидация и нормализация пути
    real_folder = os.path.realpath(folder)
    if not os.path.isdir(real_folder):
        raise ValueError(f"Указанный путь не является директорией: {folder}")
        
    # 2. Безопасный путь для архива (пример)
    backup_dir = "/var/backups"
    os.makedirs(backup_dir, exist_ok=True)
    output_path = os.path.join(backup_dir, "backup.tgz")
    
    # 3. Запуск без shell, списком аргументов
    result = subprocess.run(
        ["tar", "-czf", output_path, real_folder],
        check=False
    )
    # 4. Прямой и корректный код возврата
    return result.returncode
```
  
#### Итоговые комментарии ревьюера (PR-style)
- 🔴 **blocker:** Подстановка `folder` в команду без экранирования создаёт критический вектор Shell-инъекции. → Риск: выполнение произвольного кода от имени процесса. → Исправить: заменить `os.system` на `subprocess.run(["tar", "-czf", "backup.tgz", folder], check=True)`.
- 🟠 **major:** `os.system` скрывает детали ошибок архивации. → Риск: бэкапы могут создаваться частично или с ошибками, а функция вернёт `0`. → Исправить: использовать `subprocess.run` с проверкой `returncode` и логированием `stderr`.
- 🟡 **minor:** Отсутствует предпроверка пути. → Риск: лишний запуск `tar` с ожидаемым падением. → Исправить: добавить `os.path.isdir()` или `pathlib` проверку перед вызовом.
- 🔵 **info:** Предложение перейти на `shutil` требует согласования с архитекторами, так как меняет формат архива. → Оставляем на этапе оптимизации, не в этом PR.

#### Next Steps
- **До merge:** убрать `os.system`, перейти на `subprocess.run` со списком аргументов, добавить `check=True` и базовую валидацию пути.
- **В отдельную задачу:** перевод на асинхронные бэкапы через `asyncio.subprocess`, если процесс блокирует event-loop.
- **Тесты:** добавить unit-тест на `subprocess.run` с моком, проверить поведение при несуществующей папке и при вводе строк с спецсимволами (`;`, `&`, `$()`).
  
## Пример 2 (Java): риск NPE и неявная логика ошибок  
Ситуация: сервис отправляет письмо пользователю. Репозиторий может вернуть null  
### Исходный код
```java
public class MailService {  
    private final UserRepository userRepository;  
    private final MailClient mailClient;  

    public MailService(UserRepository userRepository, MailClient mailClient) {    
        this.userRepository = userRepository;    
        this.mailClient = mailClient;  
    }  

    public void sendWelcomeEmail(Long userId) {    
        User user = userRepository.findById(userId);     
        mailClient.send(user.getEmail(), "Welcome!");  
    }
}
```
  
### Промпт к ИИ-ревью
```
Ты выступаешь в роли Senior Java-разработчика. Ситуацияя: сервис отправки приветственного письма. Репозиторий может вернуть null, если пользователь не найден.
Необходимо провести проверки с учетом следующих критериев:
- безопасность типов (NPE)
- обработка ошибок
- читаемость
- тестируемость.  
При проверках необъодимо придерживаться следующих ограничений: не предлагай полный рефакторинг на Spring Events/MQ без причины, не выдумывай фреймворки, фокусируйся на стандартных практиках Java 8+.
Формат ответа: пронумеруй каждое замечание и укажи: blocker/major/minor, что не так, риск, как исправить. Сделай таблицу по полям: | № | Замечание ИИ | Тип | Светофор | Приоритет | Что делаем дальше |.
```

### Ответ ИИ
1. `[blocker]` `userRepository.findById(userId)` может вернуть `null`. Вызов `user.getEmail()` выбросит `NullPointerException`. **Риск:** падение сервиса при любом несуществующем ID. **Как исправить:** использовать `Optional<User>` или явную проверку `if (user == null)`.
2. `[major]` `mailClient.send()` не обрабатывается. Если SMTP недоступен, исключение улетит в caller без логирования. **Риск:** потеря писем, скрытые ошибки инфраструктуры. **Как исправить:** обернуть в try-catch, залогировать ошибку, вернуть статус или кинуть `BusinessException`.
3. `[minor]` Метод возвращает `void`. Caller не знает, успешно ли отправлено письмо. **Риск:** сложность отладки и тестирования. **Как исправить:** вернуть `boolean` или `CompletableFuture<Void>`.
4. `[doubtful]` Предлагает добавить `@Async` и `@Retryable` (Spring). **Риск:** меняет модель выполнения, требует настройки пула потоков и конфигурации重试. **Как исправить:** вынести в отдельную задачу, если требуется асинхронность.

#### Таблица классификации замечаний
| № | Замечание ИИ | Тип | Светофор | Приоритет | Что делаем дальше |
|---|--------------|-----|----------|-----------|-------------------|
| 1 | Возможный NPE при `user.getEmail()` | Корректное | Accept | blocker | Добавить проверку `if (user == null)` или `Optional.orElseThrow()`. |
| 2 | Отсутствие обработки ошибок `mailClient.send` | Корректное | Accept | major | Добавить try-catch, логирование, кастомное исключение или retry-логику. |
| 3 | `void` скрывает результат | Сомнительное | Verify | minor | Обсудить с продуктом: нужно ли возвращать статус отправки? Если да, сменить сигнатуру. |
| 4 | `@Async` + `@Retryable` | Сомнительное | Verify | minor | Проверить требования к производительности. Если синхронность приемлема, отложить. |

#### Итоговые комментарии ревьюера (PR-style)
- 🔴 **blocker:** `findById` может вернуть `null`, что приведёт к `NPE` на строке `user.getEmail()`. → Риск: 500 ошибка при отправке письма несуществующему пользователю. → Исправить: `User user = userRepository.findById(userId).orElseThrow(() -> new UserNotFoundException(userId));`.
- 🟠 **major:** Вызов `mailClient.send()` не обёрнут в обработку исключений. → Риск: сетевые ошибки SMTP превращаются в неочевидные краши без логирования. → Исправить: добавить `try-catch`, залогировать `ERROR`, выбросить `MailSendException` или вернуть статус.
- 🟡 **minor:** `void` не даёт вызывающему коду понять результат. → Риск: усложняет интеграционные тесты и бизнес-логику. → Исправить: вернуть `boolean` или `enum EmailStatus`.
- 🔵 **info:** Асинхронная отправка полезна, но требует настройки `TaskExecutor`. Выносим в бэклог.

#### Next Steps
- **До merge:** Добавить проверку на `null` / `Optional`, обернуть вызов mail-клиента в обработку ошибок, добавить логирование.
- **В отдельную задачу:** Перевод отправки писем в асинхронную очередь (Kafka/RabbitMQ или Spring `@Async`) для повышения отзывчивости API.
- **Тесты:** Unit-тест на `NullPointerException` (или `UserNotFoundException`), тест на обработку исключения от `mailClient`, mock `UserRepository` с возвратом `Optional.empty()`.
  
##  Пример 3 (Go): SQL-инъекция через конкатенацию  
Ситуация: обработчик принимает параметр status и собирает SQL строкой.  
   
```go
package handlers
import (
	"database/sql"
	"fmt"
	"net/http"
)

func Orders(db *sql.DB, w http.ResponseWriter, r *http.Request) error {
	status := r.URL.Query().Get("status")
	query := fmt.Sprintf("SELECT id, total FROM orders WHERE status = '%s'", status)
	_, err := db.Query(query)
	return err
}
```

### Промпт к ИИ-ревью
```
Ты выступаешь в роли Senior Go-разработчика. Ситуация: обработчик принимает параметр status и собирает SQL строкой. Параметр status берётся из query-строки. Проведи проверку кода с учетом следующих критериев:
- безопасность (SQL-инъекции);
- управление ресурсами (rows.Close);
- обработка HTTP-ответов, использование context.  
Учти ограничения: не предлагай ORM, не меняй сигнатуру `http.Handler` без причины, фокусируйся на стандартной `database/sql`.
Формат ответа: пронумеруй каждое замечание и укажи следующие позиции blocker/major/minor, что не так, риск, как исправить. Сделай таблицу в формате маркдаун со следующими полями: | № | Замечание ИИ | Тип | Светофор | Приоритет | Что делаем дальше |
```

### Ответ ИИ
1. `[blocker]` Конкатенация `status` через `fmt.Sprintf` создаёт SQL-инъекцию. При вводе `' OR 1=1 --` вернутся все заказы. **Риск:** утечка данных, нарушение целостности БД. **Как исправить:** использовать параметризованный запрос `db.Query("SELECT ... WHERE status = ?", status)`.
2. `[major]` Результат `db.Query` (`*sql.Rows`) не закрывается. **Риск:** утечка соединений с БД, exhaustion connection pool. **Как исправить:** `rows, err := db.Query(...); if err != nil { return err }; defer rows.Close()`.
3. `[major]` Функция возвращает `error`, но не пишет HTTP-статус или тело ответа в `w`. **Риск:** клиент всегда получает 200 OK, даже при ошибке БД. → **Как исправить:** при ошибке вызывать `http.Error(w, err.Error(), http.StatusInternalServerError)` и `return nil`.
4. `[minor]` Нет передачи `context.Context` в запрос. **Риск:** долгие/зависшие запросы не отменяются при разрыве соединения клиентом. **Как исправить:** использовать `db.QueryContext(r.Context(), "SELECT ... WHERE status = ?", status)`.
5. `[doubtful]` Предлагает заменить на GORM/Ent. **Риск:** добавляет тяжелую зависимость, меняет стиль работы с БД. **Как исправить:** оставить `database/sql` для простых запросов.

#### Таблица классификации замечаний
| № | Замечание ИИ | Тип | Светофор | Приоритет | Что делаем дальше |
|---|--------------|-----|----------|-----------|-------------------|
| 1 | SQL-инъекция через `fmt.Sprintf` | Корректное | Accept | blocker | Заменить на параметризованный запрос с `?`. |
| 2 | Не закрыт `*sql.Rows` | Корректное | Accept | major | Добавить `defer rows.Close()` после проверки ошибки. |
| 3 | Не пишется HTTP-ответ при ошибке | Корректное | Accept | major | Добавить `http.Error` и вернуть `nil` (или кастомный статус). |
| 4 | Отсутствует `context.Context` | Сомнительное | Verify | minor | Проверить: нужен ли отмена долгих запросов в этом эндпоинте? Если да, добавить `QueryContext`. |
| 5 | Переход на ORM | Ошибочное/Неуместное | Reject | minor | Отклонить: `database/sql` достаточно, ORM не требуется. |

#### Итоговые комментарии ревьюера (PR-style)
- 🔴 **blocker:** Конкатенация `status` в SQL-строку открывает прямую SQL-инъекцию. → Риск: несанкционированный доступ ко всем записям или изменение данных. → Исправить: `db.Query("SELECT id, total FROM orders WHERE status = ?", status)`.
- 🟠 **major:** `*sql.Rows` не закрывается через `defer rows.Close()`. → Риск: исчер пула соединений под нагрузкой. → Исправить: инициализировать `rows`, проверить `err`, затем `defer rows.Close()`.
- 🟠 **major:** Ошибки БД возвращаются только как `error` из функции, HTTP-клиент не получает статус. → Риск: клиент считает запрос успешным, фронтенд не показывает ошибку. → Исправить: `if err != nil { http.Error(w, "DB error", 500); return nil }`.
- 🟡 **minor:** Нет привязки запроса к `r.Context()`. → Риск: запрос продолжает выполняться после закрытия клиентского соединения. → Исправить: `db.QueryContext(r.Context(), ...)`.

#### Next Steps
- **До merge:** Убрать `fmt.Sprintf` в SQL, добавить `?` параметризацию, закрыть `rows` через `defer`, написать обработку `err` с `http.Error`.
- **В отдельную задачу:** Вынести SQL-запросы в слой `repository`, добавить пагинацию, если заказов много.
- **Тесты:** Интеграционный тест с реальным SQLite/Postgres, проверка на SQL-инъекцию (`' OR 1=1 --`), тест на закрытие соединений (`rows.Next()` + `defer Close()`), тест на отмену запроса при `context.WithTimeout`.
